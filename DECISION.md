## 决策 #1：编程语言选择

**时间**：2026-05-25

**背景**：需要在 CLI 工具和 Web 应用之间选择，以及选择合适的编程语言来实现系统。

**选项**：
- A: Python CLI — 轻量、快速开发、AI/LLM 生态成熟（openai 官方 SDK）
- B: TypeScript/Node.js Web 应用 — 可提供 Web UI，但开发周期长
- C: Rust CLI — 性能最优，但开发效率低，LLM SDK 不成熟

**决策**：选择 Python CLI（选项 A）

**理由**：
- OpenAI 官方 Python SDK 最成熟稳定
- Python 的 dataclass + Pydantic 适合结构化数据流转
- CLI 工具满足任务描述的核心需求，Web UI 是额外复杂度
- 开发效率高，6 小时内可完成完整系统
- 跨平台支持好（Windows/Linux/macOS）

**风险**：Python CLI 对非技术用户不够友好；大型项目扫描时性能可能不如编译型语言。

**后续**：如有更多时间，可添加简单的 Web UI（FastAPI + 前端），或使用 Rich 库美化 CLI 输出。

## 决策 #2：Agent 间通信方式

**时间**：2026-05-25

**背景**：三个 Agent 需要传递数据（Issue→代码→验证结果），需要选择通信方式。

**选项**：
- A: 结构化 JSON 数据传递（每轮独立调用，通过 JSON 传递上下文）
- B: 对话式上下文传递（保留完整对话历史，后续 Agent 能看到前面的所有消息）
- C: 混合式（结构化数据 + 精简的 round summary）

**决策**：选择结构化 JSON 数据传递（选项 A）

**理由**：
- 每轮独立调用避免了上下文膨胀（token 浪费 + 注意力衰减）
- JSON 结构化数据精确、可解析、可验证
- 符合 SPEC 中 "输出必须是结构化格式" 的约束
- 更易于实现重试和错误处理
- 每个 Agent 职责清晰，输入输出类型明确定义

**风险**：Agent 无法利用前几轮的 "经验"，可能重复犯同样的错误。LLM 在新上下文中可能遗漏历史信息。

**后续**：可添加轻量级 round summary（选项 C），在每轮开始时传递上一轮的简明摘要，平衡上下文大小与信息传递。

## 决策 #3：代码修复策略

**时间**：2026-05-25

**背景**：Fixer 需要输出修复后的代码。有两种主要策略：输出完整文件 vs 输出 patch/diff。

**选项**：
- A: Fixer 输出完整修复后的文件 + 每个修改的 diff（当前方案）
- B: Fixer 只输出 diff/patch，由系统 apply 到原文件
- C: Fixer 只输出修复后的完整文件，不输出 diff

**决策**：选择 A（完整文件 + diff）

**理由**：
- 完整文件确保代码完整性，避免 patch apply 失败
- Diff 便于 Validator 审查具体修改点，也便于报告中展示
- LLM 更容易准确输出完整文件而非精确的 diff 格式
- 两者结合提供了冗余保障

**风险**：输出内容较大，token 消耗高。对于大文件可能超出 token 限制。

**后续**：可对大文件实现真正的 diff-based 修复（选项 B），减少 token 消耗。当前通过文件分块（chunking）缓解。

## 决策 #4：虚假修复检测策略

**时间**：2026-05-25

**背景**：Validator 需要识别 "虚假修复"——代码改了但问题没解决。需要决定检测策略。

**选项**：
- A: 纯 LLM 判断（让 Validator Agent 自己判断）
- B: 规则引擎 + LLM（先通过规则检测常见模式，LLM 补充判断）
- C: 静态分析 + LLM（用 linter/静态分析工具先检测，LLM 做最终判断）

**决策**：选择 A（纯 LLM 判断），通过精心设计的 prompt 引导

**理由**：
- 任务明确要求 Validator 必须有此能力，但未要求具体实现方式
- 纯 LLM 方案实现简单，6 小时内可行
- Prompt 中明确定义了虚假修复的典型模式，能引导 LLM 做出正确判断
- 不需要引入额外工具依赖

**风险**：LLM 判断可能不准确，对复杂逻辑的虚假修复可能漏判。不同 LLM 模型表现差异大。

**后续**：可引入选项 B（规则引擎），对简单模式（只改注释、变量重命名等）先做规则过滤，提高准确率和效率。

## 决策 #5：配置管理方式

**时间**：2026-05-25

**背景**：系统需要管理 API Key、模型参数、超时等配置。需要选择配置方式。

**选项**：
- A: YAML 配置文件 + 环境变量 + CLI 参数（当前方案）
- B: 纯环境变量（12-factor app 风格）
- C: 纯 CLI 参数

**决策**：选择 A（三层配置：YAML + 环境变量 + CLI 参数）

**理由**：
- YAML 文件适合存储复杂配置（每个 Agent 的不同参数）
- 环境变量适合敏感信息（API Key）和 CI/CD 场景
- CLI 参数适合临时覆盖，方便调试
- 优先级：CLI > 环境变量 > YAML > 默认值，逻辑清晰

**风险**：配置来源多，调试时可能困惑哪个值生效。需要在日志中打印最终生效的配置。

**后续**：添加 `--show-config` 参数打印最终生效的完整配置。

## 决策 #6：大文件处理策略

**时间**：2026-05-25

**背景**：LLM 有 token 限制，大文件可能超过上下文窗口。需要处理策略。

**选项**：
- A: 文件分块（chunking）+ 按行号范围切分（当前方案）
- B: 截断文件（只取前 N 字符）
- C: 拒绝处理大文件

**决策**：选择 A（文件分块）

**理由**：
- 分块比截断更完整，不会丢失代码
- 分块之间可能丢失跨块上下文，但比完全不审查好
- 系统会提示用户文件被分块处理

**风险**：跨块的关联问题可能遗漏（如函数定义在一个块、调用在另一个块）。行号在分块后会偏移。

**后续**：可引入语义分块（按函数/类边界切分），或使用支持更长上下文的模型。

## 决策 #7：动态优先级实现

**时间**：2026-05-25

**背景**：任务描述中动态优先级标记为 [可选]。需要决定是否实现。

**选项**：
- A: 实现 — 每轮按 severity 排序修复顺序
- B: 不实现 — 按 Reviewer 输出顺序修复
- C: 完整实现 — 综合考虑 severity、依赖关系、修复难度

**决策**：选择 A（简单实现：按 severity 排序）

**理由**：
- 性价比高：severity 排序只需 3 行代码，却有实际价值
- 确保 critical 问题优先修复
- 选项 C 需要 LLM 评估修复难度，增加复杂度和不确定性

**风险**：简单的 severity 排序可能忽略问题间的依赖关系（修复 A 可能影响 B）。

**后续**：如有需要可扩展为拓扑排序（考虑依赖关系）。
